## v1.1.2 — Hotfix: HTTP-only deployments + login form label

Two bugs found by an operator running `:v1.1.1` on Unraid behind a
plain-HTTP IP. v1.1.2 fixes both. No schema bump, no breaking API.

### Bugs fixed

1. **Session cookie was unusable over HTTP.** The session cookie was
   set with `Secure` flag and the `__Host-` name prefix (which
   REQUIRES Secure). Every modern browser silently rejects such
   cookies on plain HTTP — login succeeded server-side
   (`login_succeeded` in audit log) but the browser dropped the
   Set-Cookie, every follow-up request 401'd, and the WebSocket
   handshake 403'd. Affected any deployment NOT behind a TLS
   reverse proxy.

2. **Login form placeholder said "Master password"** but
   `/api/auth/login` expects the **admin password** (generated by
   the container on first boot and printed to stdout via
   `RESTREAM_PLUS FIRST BOOT`, OR pre-set via `RESTREAM_ADMIN_PASSWORD`).
   Operators tried their `RESTREAM_MASTER_PASSPHRASE` (the at-rest
   key-derivation passphrase, NOT a login credential) and got
   "wrong password."

### What changed

- **New env var `RESTREAM_COOKIE_SECURE`** (`AppSettings.cookie_secure`,
  default `true`). When `true` the cookie name is `__Host-rp_session`
  with the `Secure` flag — strongest stock posture, requires the
  server to be reachable over HTTPS (a TLS reverse proxy in front
  of the container is the supported production shape). When `false`
  the cookie name drops the `__Host-` prefix to `rp_session` and
  the `Secure` flag is omitted — works on plain HTTP, retains
  `HttpOnly` + `SameSite=Lax` + `Path=/`.
- `app/auth/sessions.py::SESSION_COOKIE_NAME` (constant) is now a
  function `session_cookie_name(*, secure: bool)`. `build_cookie_set_kwargs`
  and `build_cookie_delete_kwargs` now take `secure: bool`. All four
  call sites (`app/api/auth.py` × 4, `app/api/security_api.py` × 2,
  `app/api/ws.py` × 1, `app/auth/deps.py` × 1) thread the setting
  through. Existing `SESSION_COOKIE_NAME` constant kept as alias
  for `__Host-rp_session` so external code that read it still
  works in secure mode.
- **Login form placeholder fixed**: "Master password" → "Admin password"
  ([web/src/messages.ts](../../web/src/messages.ts)), with an
  inline code comment naming the v1.1.1 confusion for future readers.
- **2 new tests** in `tests/auth/test_sessions.py` pin the
  insecure-mode cookie posture (name drops `__Host-`, Secure flag
  omitted; other defenses retained). Existing test suite (669 tests)
  refactored to use `cookie_secure=False` by default since the test
  HTTP client runs over `http://` and previously needed a manual
  cookie-injection workaround in conftest.

### Upgrade

For **plain-HTTP setups** (LAN, NAS, direct-IP), add
`-e RESTREAM_COOKIE_SECURE=false`:

```bash
docker pull ghcr.io/pelmentor/restream-plus:v1.1.2
docker stop -t 30 restream-plus
docker rm restream-plus
docker run -d \
  --name restream-plus \
  --restart unless-stopped \
  -p 1935:1935 -p 8000:8000 \
  -v /mnt/user/appdata/restream-plus:/data \
  --stop-timeout 30 \
  -e RESTREAM_MASTER_PASSPHRASE='same-passphrase-as-before' \
  -e RESTREAM_COOKIE_SECURE=false \
  ghcr.io/pelmentor/restream-plus:v1.1.2
```

For **HTTPS / TLS-reverse-proxy setups** (production), DO NOT set
`RESTREAM_COOKIE_SECURE` — the default `true` keeps the strongest
posture:

```bash
docker pull ghcr.io/pelmentor/restream-plus:v1.1.2
# ... otherwise the same docker run command as v1.1.1, no new envs.
```

Volume + DB + encrypted credentials persist across the recreate.
**Existing logged-in sessions invalidate** because the cookie name
changes (`__Host-rp_session` ↔ `rp_session` depending on mode); log
in again with your admin password.

### Breaking changes vs. v1.1.1

None to the API. The cookie name change does log out existing
sessions once on upgrade (the browser sees a different cookie name
in the Set-Cookie response and the old one becomes orphan); this is
the only operator-visible change.

### Postmortem note

Both bugs are operator-honest "I tried to use it and it didn't
work." Neither was covered by automated tests because:
1. The HTTP-cookie issue was masked in the test suite by a manual
   cookie-injection workaround in `conftest.py` (added years ago
   when the same problem surfaced in tests but was never fixed in
   production). The workaround is now gone — tests use the new
   `cookie_secure=False` mode and httpx persists the cookie
   naturally.
2. The login form label was correct *semantics* per ADR-0005 but
   used the wrong *word*. No test asserts the literal placeholder
   text.

`release-checklist.md` step 12a (browser reachability, added in
v1.1.1) MUST now include an actual login attempt with the admin
password from `docker logs … FIRST BOOT`. Updating that next.
