## v1.1.3 — Hotfix: SPA relative-route loop crashed backend

Operator reported "internal server error after clicking around the
panel" on `:v1.1.2`. v1.1.3 fixes the bug at both the layer that
caused it (frontend) and the layer that crashed because of it
(backend). No schema bump, no breaking API.

### Bug

Clicking certain nav targets in `/settings/*` could navigate to URLs
like
`/settings/targets/twitch/targets/twitch/general/general/general/…`
— each click appending another segment. Once the path exceeded
ext4's 4096-byte filename limit, the SPA fallback in `app/main.py`
crashed with `OSError: [Errno 36] File name too long` inside
`Path.is_file()`, returning HTTP 500.

### Root cause

`web/src/pages/settings/index.tsx` used **relative** `Navigate to=…`
targets — most critically inside `<Route path="*">`, the catch-all
for unmatched nested routes. React Router resolved the relative
target against the splat-matched URL, so the redirect appended a
new segment rather than replacing the path:

| URL                           | `<Navigate to="general">` resolves to       |
| ----------------------------- | ------------------------------------------- |
| `/settings/foo`               | `/settings/foo/general`                     |
| `/settings/foo/general`       | `/settings/foo/general/general`             |
| `/settings/foo/general/general` | `/settings/foo/general/general/general` … |

The browser kept following the redirects until the URL exceeded the
OS path-length limit, at which point the backend `_spa_fallback`
called `Path.is_file()` on a >4 KB path and crashed.

### What changed

- **Frontend (root cause)**: every `Navigate` / `NavLink` `to=…` in
  the settings surface now uses an absolute path
  (`/settings/general`, `/settings/targets/twitch`, …). Inline code
  comments in `web/src/pages/settings/index.tsx` and
  `web/src/components/settings/SettingsSidebar.tsx` name the
  compounding-loop hazard so future contributors don't reintroduce
  relative routes.
- **Backend (defense in depth)**: `app/main.py::_spa_fallback`
  - caps `full_path` length at 1024 chars (well under ext4's 4096)
    and returns `index.html` for anything longer, short-circuiting
    before any `stat()` syscall;
  - catches `OSError`/`ValueError` from `Path.resolve()` and returns
    a real **404** (not the SPA shell) — a path the OS can't even
    resolve is not a legitimate deep link and keeping it distinct
    from happy-path navigations preserves the v1.1.1 invariant that
    "unresolvable / traversal-shaped requests return 404";
  - catches `OSError`/`ValueError` from `Path.is_file()` and falls
    through to `index.html` — at that point the path is verified
    in-bounds, so serving the SPA shell is safe;
  - **never crashes the request**, regardless of input. The path-
    traversal explicit 404 (introduced in v1.1.1) is preserved for
    URLs that resolve outside the SPA mount root.
- **3 new tests** in `tests/api/test_spa_serving.py`:
  - `test_extremely_long_path_does_not_crash` — `/settings/general/`
    repeated 600× (~4800 chars) must NOT return 500.
  - `test_path_with_unstattable_segments_does_not_crash` — a
    URL-encoded NUL byte (`%00`) must NOT return 500 (this caught
    a Windows-vs-POSIX exception class mismatch during dev:
    `Path.resolve()` raises `ValueError` on embedded NULs, not
    `OSError`).
  - `test_resolve_oserror_is_caught_and_returns_404` — pins the
    `resolve()` guard distinctly from the `is_file()` guard, so a
    future refactor that moves one but not the other regresses
    loudly instead of silently.

### Breaking changes vs. v1.1.2

None. Backend response semantics unchanged for all happy-path
requests; only the formerly-crashing pathological inputs now return
`index.html` instead of HTTP 500. Frontend nav targets work
identically — the URLs themselves are unchanged (`/settings/general`
is still `/settings/general`), only the link `to=` source values
became absolute.

### Postmortem note

Same class of bug as v1.1.1 / v1.1.2: a "you only see it when you
click around in a real browser" issue that automated tests didn't
catch. The frontend test suite has no router-integration coverage
(no Playwright, no `@testing-library/react-router-dom` exercise
spanning multiple navigations). Worth adding before v1.2.0.
